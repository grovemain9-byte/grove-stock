## AI組織 — 6部門

### 1. 執行部門
**ミッション:** 24/7でルールに従い売買を実行し続ける

- 市場スキャン（30分ごと）
- 5プレイヤー投票による売買判断
- 注文執行・ポジション管理
- 全判断値をDuckDBに記録（RSI実値・BB距離・出来高等）

### 2. リスク管理部門
**ミッション:** 損失をコントロールし組織を守る

- 月次ドローダウン監視（-10%で自動停止）
- ポジション単体の損切り管理（-5%）
- リスク指標の継続モニタリング

### 3. 調査・研究部門
**ミッション:** 実績データを分析して改善提案を行う（Phase 2）

- 週次: 執行部門の成績を複数LLMで分析
- Type A: パラメータ改善提案（閾値変更等）
- Type B: 新戦略・新プレイヤー提案（P6追加等）
- 複数LLMを使う理由: 単一LLMのバイアスを避けるため

### 4. 財務・経理部門
**ミッション:** お金の流れを正確に記録・報告する

- 取引PnLの記録
- 月次損益計算・決算
- 資金の出し入れ管理（追加入金・利益出金）
- 税務計算（雑所得・円換算）
- 将来: 外部資金管理（※金融商品取引業登録が必要になる）

### 5. エンジニアリング部門
**ミッション:** 組織を技術的に進化させる（Phase 2）

- 調査・研究部門の提案をGitHub Issueに変換
- grove-frameworkのハーネスでClaude Codeを起動
- 人間承認 → マージ → grove-stockが進化する
- コードの自律的進化を担う

### 6. ガバナンス部門
**ミッション:** 組織の透明性と継続性を担保する

- 週次レポート作成（取締役会向け）
- 月次決算レポート
- MTG資料・議事録
- 各部門の活動ログ管理

## フェーズ設計

### Phase 1（今作る）
執行部門 + リスク管理部門の基本機能

```
24/7稼働
├── Ingestion → Voting → Kelly → Execution → Monitor → Exit
├── 全判断値をDuckDBに記録
└── 月次ドローダウン-10%で自動停止
```

### Phase 2（実績が溜まってから）
残り4部門を順次立ち上げる

```
調査・研究部門が週次で分析・提案
    ↓
エンジニアリング部門がGitHub Issue生成
    ↓
grove-frameworkのハーネスがClaude Codeを動かす
    ↓
人間承認 → マージ
    ↓
grove-stockが進化する
```

## 各部門のエージェント構成
> 2026-04-07確定

| 部門 | 実体 | 使用ツール |
|------|------|---------|
| 執行部門 | 5ノード（ingestion / scan / vote / kelly / execution） | LangGraph scan_graph・pandas-ta・J-Quants API |
| リスク管理部門 | 4ノード（drawdown / stopper / emergency / macro） | LangGraph monitor_graph・Grok API（日次） |
| 調査・研究部門 | Phase 2 | 複数LLM（Claude / Grok / Qwen）|
| 財務・経理部門 | Phase 2 | DuckDB + Python |
| エンジニアリング部門 | Phase 2 | Claude Code + GitHub Actions |
| ガバナンス部門 | Phase 2 | Claude API |

**全ノード共通:** ルールベース（LLMなし）。例外はリスク管理部門のmacroノード（Grok API で日次マクロ検知）のみ。
